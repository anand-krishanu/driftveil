# DriftVeil Production Database Design

This plan outlines the relational database architecture required to migrate the DriftVeil backend from static CSV/JSON files (`sensor_stream.csv` and `fingerprints.json`) to a production-ready database (e.g., PostgreSQL with TimescaleDB for time-series data).

## User Review Required

> [!IMPORTANT]
> **Database Choice:** This schema assumes **PostgreSQL with the TimescaleDB extension**. TimescaleDB is the industry standard for handling high-frequency SCADA/IoT time-series data. Are you comfortable targeting PostgreSQL/TimescaleDB, or would you prefer a generic SQL (MySQL) or NoSQL (MongoDB) approach?

## Proposed Schema

The database is divided into four core areas: **Assets**, **Telemetry**, **AI Intelligence**, and **Operations**.

### 1. Asset Registry (`machines`)
Stores the static metadata for the industrial equipment being monitored.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Primary Key | Unique identifier for the machine |
| `name` | `VARCHAR(100)` | Not Null | E.g., "Bottling Unit C" |
| `line` | `VARCHAR(50)` | | Production line assignment |
| `location` | `VARCHAR(100)`| | Physical location (e.g., "Bay B-1") |
| `base_health` | `INTEGER` | Default 100 | Baseline health score without drift |
| `status` | `VARCHAR(20)` | Default 'NORMAL'| Current aggregated status (`NORMAL`, `WARN`, `DRIFT`) |
| `installed_at`| `TIMESTAMPTZ` | | Date of asset installation |
| `created_at` | `TIMESTAMPTZ` | Default NOW() | Record creation timestamp |

---

### 2. Time-Series Telemetry (`sensor_readings`)
This will be a TimescaleDB Hypertable. It captures the high-frequency metrics pushed by the PLC/Edge devices.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `time` | `TIMESTAMPTZ` | Primary Key (Composite) | Exact time of the reading |
| `machine_id` | `UUID` | Primary Key, F-Key | References `machines(id)` |
| `temperature` | `FLOAT` | | Temperature reading in °C |
| `vibration` | `FLOAT` | | Vibration reading in mm/s |
| `rpm` | `INTEGER` | Optional | Contextual data (if applicable) |

> [!TIP]
> **Hypertable Partitioning:** In TimescaleDB, this table will be partitioned by `time` (e.g., 1-day chunks) and indexed heavily on `machine_id` to allow millisecond lookups when the AI Monitor Agent requests the "last 15 rows".

---

### 3. AI Intelligence (`failure_fingerprints`)
Replaces `fingerprints.json`. Stores the historical and physics-based failure modes that Agent 3 uses for Root Cause matching.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Primary Key | Unique identifier |
| `issue_name` | `VARCHAR(150)`| Not Null | E.g., "Early Bearing Wear" |
| `severity` | `VARCHAR(20)` | Not Null | `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` |
| `pattern_data`| `JSONB` | | Structured signature (e.g., {"temp_slope": ">0.1", "vib": ">0.05"}) |
| `eta_days` | `INTEGER` | | Average days to failure after signature match |
| `action_prescription`| `TEXT` | | Standard operating procedure for mitigation |

---

### 4. Operations & Alerts (`alerts`)
Whenever Agents 2, 3, and 4 detect and diagnose a drift, the result is saved here for the Operator and Engineer Views.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Primary Key | Alert ID (e.g., DV-2024-0847) |
| `machine_id` | `UUID` | F-Key | References `machines(id)` |
| `timestamp` | `TIMESTAMPTZ` | Default NOW() | When the drift was detected |
| `title` | `VARCHAR(255)`| | Plain English AI Title |
| `severity` | `VARCHAR(20)` | | Severity generated by Agent 4 |
| `confidence` | `INTEGER` | | Percentage (0-100) generated by Agent 3 |
| `eta_days` | `INTEGER` | | Projected failure timeframe |
| `action` | `TEXT` | | Prescribed action string |
| `raw_diagnosis`| `TEXT` | | The raw explanation block from Agent 3 |
| `cusum_score` | `FLOAT` | | The S_t math score at time of detection |
| `resolved` | `BOOLEAN` | Default FALSE | Whether maintenance has cleared the alert |

---

## Open Questions

1. **ORM Choice:** Will the backend use an ORM (like SQLAlchemy for Python, or Hibernate/JPA if migrating to Spring Boot) or will we write raw SQL queries for performance?
2. **Historical Retention:** How long do we need to retain `sensor_readings`? (TimescaleDB allows configuring automatic retention policies like "drop data older than 30 days").
